# Единый архитектурный документ (задания 7 - 10)

## Задание 7. Проектирование схем коллекций для шардирования данных

### 1. Введение

Данный документ описывает проектирование схем коллекций MongoDB для онлайн-магазина «Мобильный мир». Рассматриваются три ключевые коллекции: `products`, `orders`, `carts`. Для каждой коллекции определены структура данных, стратегия шардирования и обоснование выбора шард-ключа.

### 2. Коллекция products

#### 2.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор товара
  name: String,                     // Наименование товара
  category: String,                 // Категория товара (индексированное поле)
  price: Decimal,                   // Цена товара (используем Decimal для точности)
  stock: {                          // Остаток товара по геозонам
    "Moscow": Number,
    "Saint-Petersburg": Number,
    "Ekaterinburg": Number,
    "Kaliningrad": Number,
    // ... другие геозоны
  },
  attributes: {                     // Дополнительные атрибуты (гибкая структура)
    color: String,
    size: String,
    brand: String,
    // ... другие атрибуты
  },
  created_at: Date,                 // Дата создания
  updated_at: Date                  // Дата последнего обновления
}
```

#### 2.2. Индексы
```
// Для поиска по категориям и фильтрации по цене
db.products.createIndex({ category: 1, price: 1 });

// Для поиска по имени (текстовый поиск)
db.products.createIndex({ name: "text" });

// Для быстрого получения товара по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 2.3. Стратегия шардирования
**Шард-ключ:** `{ _id: "hashed" }` (хешированное шардирование)
**Тип шардирования:** Hashed Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Отлично — хеширование обеспечивает равномерное распределение документов по шардам |
| **Write Scaling** | Отлично — вставки новых товаров распределяются равномерно |
| **Query Pattern** | Большинство запросов — по _id (страница товара) или по категориям с фильтрацией |
| **Read Scaling** | Поиск по категориям требует broadcast queries, но это приемлемо |

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Broadcast queries** | Поиск по категориям и цене может отправляться на все шарды | Использовать композитные индексы, рассмотреть возможность денормализации |
| **Hot spots** | При использовании hashed шардирования отсутствуют | Нет риска |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать хешированный индекс
db.products.createIndex({ _id: "hashed" });

// Шардировать коллекцию
sh.shardCollection("somedb.products", { _id: "hashed" });
```

### 3. Коллекция orders

#### 3.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор заказа
  user_id: ObjectId,                // Идентификатор клиента
  order_date: Date,                 // Дата и время оформления заказа
  items: [                          // Список заказанных товаров
    {
      product_id: ObjectId,
      product_name: String,         // Денормализовано для истории
      price: Decimal,               // Цена на момент покупки
      quantity: Number
    }
  ],
  status: String,                   // Статус заказа: "pending", "paid", "shipped", "delivered", "cancelled"
  total_amount: Decimal,            // Общая сумма заказа
  geo_zone: String,                 // Геозона заказа (например, "Moscow")
  created_at: Date,                 // Дата создания
  updated_at: Date                  // Дата последнего обновления
}
```

#### 3.2. Индексы
```
// Для поиска истории заказов пользователя
db.orders.createIndex({ user_id: 1, order_date: -1 });

// Для поиска по статусу и геозоне (операционные запросы)
db.orders.createIndex({ status: 1, geo_zone: 1 });

// Для быстрого получения заказа по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 3.3. Стратегия шардирования
**Шард-ключ:** `{ user_id: "hashed" }` (хешированное шардирование)
**Тип шардирования:** Hashed Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Отлично — хеширование user_id равномерно распределяет заказы пользователей |
| **Write Scaling** | Отлично — новые заказы равномерно распределяются по шардам |
| **Query Pattern** | Основные запросы: по user_id (история заказов) и по _id (статус) |
| **Read Scaling** | Запросы по user_id попадают в один шард (локальные) |

Преимущества выбора user_id как шард-ключа:
1. Локальность запросов — история заказов пользователя хранится в одном шарде, что обеспечивает быстрый доступ
2. Равномерное распределение — хеширование user_id дает равномерное распределение нагрузки
3. Оптимальность для создания заказов — новые заказы равномерно распределяются

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Hot users** | У популярных пользователей может быть много заказов | Хеширование распределяет заказы равномерно, т.к. разные заказы одного пользователя имеют разные _id, но user_id одинаковый — все заказы одного пользователя попадают в один шард |
| **Монотонный рост** | При использовании hashed нет проблемы монотонного роста | Нет риска |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать хешированный индекс
db.orders.createIndex({ user_id: "hashed" });

// Шардировать коллекцию
sh.shardCollection("somedb.orders", { user_id: "hashed" });
```

### 4. Коллекция carts

#### 4.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор корзины
  user_id: ObjectId,                // Идентификатор пользователя (опционально)
  session_id: String,               // Session ID для гостей (опционально)
  items: [                          // Список товаров в корзине
    {
      product_id: ObjectId,
      quantity: Number
    }
  ],
  status: String,                   // Статус корзины: "active", "ordered", "abandoned"
  created_at: Date,                 // Дата и время создания
  updated_at: Date,                 // Дата и время последнего обновления
  expires_at: Date                  // Время удаления (TTL)
}
```

#### 4.2. Индексы
```
// Для поиска активной корзины по пользователю
db.carts.createIndex({ user_id: 1, status: 1 }, { partialFilterExpression: { status: "active" } });

// Для поиска активной корзины по сессии
db.carts.createIndex({ session_id: 1, status: 1 }, { partialFilterExpression: { status: "active" } });

// TTL индекс для автоматической очистки
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 });

// Для быстрого получения корзины по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 4.3. Стратегия шардирования
**Шард-ключ:** `{ status: 1, user_id: 1 }` (диапазонное шардирование с композитным ключом)
**Тип шардирования:** Ranged Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Хорошее — активные корзины (status="active") могут быть распределены по user_id |
| **Write Scaling** | Хорошее — обновления корзин распределяются по user_id |
| **Query Pattern** | Поиск всегда идет по { user_id, status:"active" } или { session_id, status:"active" } |
| **Read Scaling** | Запросы по user_id попадают в один шард (локальные) |

Почему не hashed:
1. Ключевая особенность — корзины имеют разный статус, и большинство операций происходит только с активными корзинами
2. Префикс шард-ключа status — позволяет изолировать данные по статусу
3. Композитный ключ status, user_id — обеспечивает локальность запросов для активных корзин

Схема распределения:
```
Shard 1: status="active", user_id: MinKey → user_id: 5fffffff...
Shard 2: status="active", user_id: 60000000... → MaxKey
Shard 3: status="ordered", все user_id
Shard 4: status="abandoned", все user_id
```

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Jumbo chunks** | Статус "abandoned" может накопить много данных | Использовать TTL для автоматической очистки старых корзин |
| **Zone sharding** | Активные корзины требуют больше ресурсов | Рассмотреть возможность zone sharding для изоляции активных корзин |
| **Обновление статуса** | При смене статуса документа может потребоваться перемещение между шардами | Принимаем это как компромисс — операция происходит редко |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать индекс для шардирования
db.carts.createIndex({ status: 1, user_id: 1 });

// Шардировать коллекцию
sh.shardCollection("somedb.carts", { status: 1, user_id: 1 });

// Настроить зоны для изоляции активных корзин (опционально)
sh.addShardToZone("shard1ReplSet", "active_zone");
sh.addShardToZone("shard2ReplSet", "active_zone");
sh.updateZoneKeyRange("somedb.carts", { status: "active", user_id: MinKey() }, 
                                    { status: "active", user_id: MaxKey() }, "active_zone");
```

### 5. Сводная таблица стратегий шардирования
| Коллекция | Шард-ключ | Тип | Обоснование |
|-----------|-----------|-----|-------------|
| **products** | `{ _id: "hashed" }` | Hashed | Равномерное распределение вставок, поиск по _id локален |
| **orders** | `{ user_id: "hashed" }` | Hashed | Локальность запросов истории заказов, равномерное распределение |
| **carts** | `{ status: 1, user_id: 1 }` | Ranged | Изоляция активных корзин, локальность запросов по пользователю |

### 6. Операционные сценарии

#### 6.1. Создание заказа
1. Найти активную корзину пользователя
2. Проверить остатки товаров (products.stock)
3. Обновить остатки (decrement stock)
4. Создать документ в orders
5. Обновить статус корзины на "ordered"
6. Очистить кэш Redis

#### 6.2. Слияние гостевой корзины
1. Найти активную гостевую корзину по session_id
2. Найти активную пользовательскую корзину по user_id
3. Объединить items (суммировать quantity)
4. Обновить пользовательскую корзину
5. Отметить гостевую корзину как "abandoned"
6. Очистить кэш Redis

### 7. Метрики мониторинга
| Метрика | Описание | Действие при проблеме |
|---------|----------|----------------------|
| **Shard balance** | Разница в количестве документов между шардами (> 20%) | Запустить балансировщик: `sh.startBalancer()` |
| **Query time** | Время выполнения запросов (> 500ms) | Проверить индексы, оптимизировать запросы |
| **Write latency** | Задержка записи (> 100ms) | Проверить нагрузку на PRIMARY реплики |
| **Connection pool** | Количество соединений к mongos | Масштабировать mongos |
| **Chunk distribution** | Количество chunk'ов на шард | Проверить балансировку |
| **Jumbo chunks** | Chunk'и, превышающие лимит | Разбить вручную или увеличить лимит |
| **TTL cleanup** | Эффективность очистки старых корзин | Проверить TTL индекс |

#### 7.1. Команды для мониторинга
```
// Проверить баланс шардов
sh.status()

// Проверить распределение chunk'ов
db.getSiblingDB("config").chunks.aggregate([
  { $group: { _id: "$shard", count: { $sum: 1 } } }
])

// Проверить время выполнения запросов
db.setProfilingLevel(2)  // Включить slow query logging

// Проверить статус балансировки
sh.getBalancerState()
```

### 8. Заключение
Предложенные схемы коллекций и стратегии шардирования обеспечивают:
1. Масштабируемость — равномерное распределение данных по шардам
2. Производительность — локальность запросов для основных операций
3. Отказоустойчивость — репликация для каждого шарда
4. Оптимизацию памяти — TTL для автоматической очистки старых корзин

Ключевые решения:
- **products** — хешированное шардирование по _id для равномерной нагрузки
- **orders** — хешированное шардирование по user_id для локальности истории
- **carts** — диапазонное шардирование по { status, user_id } для изоляции активных данных
