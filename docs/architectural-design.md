# Единый архитектурный документ (задания 7 - 10)

## Задание 7. Проектирование схем коллекций для шардирования данных

### 1. Введение

Данный документ описывает проектирование схем коллекций MongoDB для онлайн-магазина «Мобильный мир». Рассматриваются три ключевые коллекции: `products`, `orders`, `carts`. Для каждой коллекции определены структура данных, стратегия шардирования и обоснование выбора шард-ключа.

### 2. Коллекция products

#### 2.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор товара
  name: String,                     // Наименование товара
  category: String,                 // Категория товара (индексированное поле)
  price: Decimal,                   // Цена товара (используем Decimal для точности)
  stock: {                          // Остаток товара по геозонам
    "Moscow": Number,
    "Saint-Petersburg": Number,
    "Ekaterinburg": Number,
    "Kaliningrad": Number,
    // ... другие геозоны
  },
  attributes: {                     // Дополнительные атрибуты (гибкая структура)
    color: String,
    size: String,
    brand: String,
    // ... другие атрибуты
  },
  created_at: Date,                 // Дата создания
  updated_at: Date                  // Дата последнего обновления
}
```

#### 2.2. Индексы
```
// Для поиска по категориям и фильтрации по цене
db.products.createIndex({ category: 1, price: 1 });

// Для поиска по имени (текстовый поиск)
db.products.createIndex({ name: "text" });

// Для быстрого получения товара по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 2.3. Стратегия шардирования
**Шард-ключ:** `{ _id: "hashed" }` (хешированное шардирование)
**Тип шардирования:** Hashed Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Отлично — хеширование обеспечивает равномерное распределение документов по шардам |
| **Write Scaling** | Отлично — вставки новых товаров распределяются равномерно |
| **Query Pattern** | Большинство запросов — по _id (страница товара) или по категориям с фильтрацией |
| **Read Scaling** | Поиск по категориям требует broadcast queries, но это приемлемо |

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Broadcast queries** | Поиск по категориям и цене может отправляться на все шарды | Использовать композитные индексы, рассмотреть возможность денормализации |
| **Hot spots** | При использовании hashed шардирования отсутствуют | Нет риска |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать хешированный индекс
db.products.createIndex({ _id: "hashed" });

// Шардировать коллекцию
sh.shardCollection("somedb.products", { _id: "hashed" });
```

### 3. Коллекция orders

#### 3.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор заказа
  user_id: ObjectId,                // Идентификатор клиента
  order_date: Date,                 // Дата и время оформления заказа
  items: [                          // Список заказанных товаров
    {
      product_id: ObjectId,
      product_name: String,         // Денормализовано для истории
      price: Decimal,               // Цена на момент покупки
      quantity: Number
    }
  ],
  status: String,                   // Статус заказа: "pending", "paid", "shipped", "delivered", "cancelled"
  total_amount: Decimal,            // Общая сумма заказа
  geo_zone: String,                 // Геозона заказа (например, "Moscow")
  created_at: Date,                 // Дата создания
  updated_at: Date                  // Дата последнего обновления
}
```

#### 3.2. Индексы
```
// Для поиска истории заказов пользователя
db.orders.createIndex({ user_id: 1, order_date: -1 });

// Для поиска по статусу и геозоне (операционные запросы)
db.orders.createIndex({ status: 1, geo_zone: 1 });

// Для быстрого получения заказа по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 3.3. Стратегия шардирования
**Шард-ключ:** `{ user_id: "hashed" }` (хешированное шардирование)
**Тип шардирования:** Hashed Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Отлично — хеширование user_id равномерно распределяет заказы пользователей |
| **Write Scaling** | Отлично — новые заказы равномерно распределяются по шардам |
| **Query Pattern** | Основные запросы: по user_id (история заказов) и по _id (статус) |
| **Read Scaling** | Запросы по user_id попадают в один шард (локальные) |

Преимущества выбора user_id как шард-ключа:
1. Локальность запросов — история заказов пользователя хранится в одном шарде, что обеспечивает быстрый доступ
2. Равномерное распределение — хеширование user_id дает равномерное распределение нагрузки
3. Оптимальность для создания заказов — новые заказы равномерно распределяются

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Hot users** | У популярных пользователей может быть много заказов | Хеширование распределяет заказы равномерно, т.к. разные заказы одного пользователя имеют разные _id, но user_id одинаковый — все заказы одного пользователя попадают в один шард |
| **Монотонный рост** | При использовании hashed нет проблемы монотонного роста | Нет риска |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать хешированный индекс
db.orders.createIndex({ user_id: "hashed" });

// Шардировать коллекцию
sh.shardCollection("somedb.orders", { user_id: "hashed" });
```

### 4. Коллекция carts

#### 4.1. Схема коллекции
```
{
  _id: ObjectId,                    // Уникальный идентификатор корзины
  user_id: ObjectId,                // Идентификатор пользователя (опционально)
  session_id: String,               // Session ID для гостей (опционально)
  items: [                          // Список товаров в корзине
    {
      product_id: ObjectId,
      quantity: Number
    }
  ],
  status: String,                   // Статус корзины: "active", "ordered", "abandoned"
  created_at: Date,                 // Дата и время создания
  updated_at: Date,                 // Дата и время последнего обновления
  expires_at: Date                  // Время удаления (TTL)
}
```

#### 4.2. Индексы
```
// Для поиска активной корзины по пользователю
db.carts.createIndex({ user_id: 1, status: 1 }, { partialFilterExpression: { status: "active" } });

// Для поиска активной корзины по сессии
db.carts.createIndex({ session_id: 1, status: 1 }, { partialFilterExpression: { status: "active" } });

// TTL индекс для автоматической очистки
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 });

// Для быстрого получения корзины по ID (автоматически)
// _id имеет уникальный индекс по умолчанию
```

#### 4.3. Стратегия шардирования
**Шард-ключ:** `{ status: 1, user_id: 1 }` (диапазонное шардирование с композитным ключом)
**Тип шардирования:** Ranged Sharding

Обоснование выбора:
| Критерий | Оценка |
|----------|--------|
| **Распределение нагрузки** | Хорошее — активные корзины (status="active") могут быть распределены по user_id |
| **Write Scaling** | Хорошее — обновления корзин распределяются по user_id |
| **Query Pattern** | Поиск всегда идет по { user_id, status:"active" } или { session_id, status:"active" } |
| **Read Scaling** | Запросы по user_id попадают в один шард (локальные) |

Почему не hashed:
1. Ключевая особенность — корзины имеют разный статус, и большинство операций происходит только с активными корзинами
2. Префикс шард-ключа status — позволяет изолировать данные по статусу
3. Композитный ключ status, user_id — обеспечивает локальность запросов для активных корзин

Схема распределения:
```
Shard 1: status="active", user_id: MinKey → user_id: 5fffffff...
Shard 2: status="active", user_id: 60000000... → MaxKey
Shard 3: status="ordered", все user_id
Shard 4: status="abandoned", все user_id
```

Риски и их mitigation:
| Риск | Описание | Mitigation |
|------|----------|------------|
| **Jumbo chunks** | Статус "abandoned" может накопить много данных | Использовать TTL для автоматической очистки старых корзин |
| **Zone sharding** | Активные корзины требуют больше ресурсов | Рассмотреть возможность zone sharding для изоляции активных корзин |
| **Обновление статуса** | При смене статуса документа может потребоваться перемещение между шардами | Принимаем это как компромисс — операция происходит редко |

Команда для включения шардирования:
```
// Включить шардирование для базы данных
sh.enableSharding("somedb");

// Создать индекс для шардирования
db.carts.createIndex({ status: 1, user_id: 1 });

// Шардировать коллекцию
sh.shardCollection("somedb.carts", { status: 1, user_id: 1 });

// Настроить зоны для изоляции активных корзин (опционально)
sh.addShardToZone("shard1ReplSet", "active_zone");
sh.addShardToZone("shard2ReplSet", "active_zone");
sh.updateZoneKeyRange("somedb.carts", { status: "active", user_id: MinKey() }, 
                                    { status: "active", user_id: MaxKey() }, "active_zone");
```

### 7. Метрики мониторинга
| Метрика | Описание | Действие при проблеме |
|---------|----------|----------------------|
| **Shard balance** | Разница в количестве документов между шардами (> 20%) | Запустить балансировщик: `sh.startBalancer()` |
| **Query time** | Время выполнения запросов (> 500ms) | Проверить индексы, оптимизировать запросы |
| **Write latency** | Задержка записи (> 100ms) | Проверить нагрузку на PRIMARY реплики |
| **Connection pool** | Количество соединений к mongos | Масштабировать mongos |
| **Chunk distribution** | Количество chunk'ов на шард | Проверить балансировку |
| **Jumbo chunks** | Chunk'и, превышающие лимит | Разбить вручную или увеличить лимит |
| **TTL cleanup** | Эффективность очистки старых корзин | Проверить TTL индекс |

### 8. Заключение
Предложенные схемы коллекций и стратегии шардирования обеспечивают:
1. Масштабируемость — равномерное распределение данных по шардам
2. Производительность — локальность запросов для основных операций
3. Отказоустойчивость — репликация для каждого шарда
4. Оптимизацию памяти — TTL для автоматической очистки старых корзин

Ключевые решения:
- **products** — хешированное шардирование по _id для равномерной нагрузки
- **orders** — хешированное шардирование по user_id для локальности истории
- **carts** — диапазонное шардирование по { status, user_id } для изоляции активных данных

## Задание 8. Выявление и устранение «горячих» шардов

### 1. Проблема: «горячие» шарды

В текущей архитектуре коллекция `products` шардирована по `{ _id: "hashed" }`. Это обеспечивает равномерное распределение документов, но **не решает проблему неравномерной нагрузки на уровне запросов**. Категория «Электроника» генерирует 70% всех запросов, что создает непропорциональную нагрузку на шарды, где хранятся эти документы.

**Корень проблемы:** Даже при равномерном распределении документов, запросы к популярным документам могут создавать «горячие» точки (hot spots) на конкретных шардах.

### 2. Метрики мониторинга для выявления «горячих» шардов

| Категория | Метрика | Описание | Порог срабатывания | Источник данных |
|-----------|---------|----------|-------------------|-----------------|
| **Нагрузка на шарды** | `shard_query_rate` | Количество запросов в секунду на шард | > 20% от общего трафика на один шард | mongos logs, `db.currentOp()` |
| | `shard_cpu_usage` | Использование CPU на узлах шарда | > 80% в течение 5 минут | Docker stats, Prometheus |
| | `shard_io_wait` | Время ожидания ввода-вывода | > 50ms | iostat, mongostat |
| | `shard_lock_percentage` | Процент времени блокировок | > 30% | `db.serverStatus().locks` |
| **Размер данных** | `chunk_size_distribution` | Размер chunk'ов по шардам | Разница > 20% | `sh.status()` |
| | `documents_per_shard` | Количество документов на шард | Разница > 20% | `sh.status()` |
| **Тепловые карты** | `hot_collection_reads` | Частота чтения коллекции по шардам | > 50% на один шард | `db.collection.getShardDistribution()` |
| | `query_pattern_by_category` | Распределение запросов по категориям | 70% на одну категорию | Логи приложения, APM |
| **Операционные** | `migration_queue_length` | Длина очереди миграций | > 10 | `sh.getBalancerState()` |
| | `balancer_runtime` | Время работы балансировщика | > 10 минут | `sh.status()` |

### 4. Механизмы устранения дисбаланса

#### 4.1. Автоматическая балансировка (встроенная)
MongoDB имеет встроенный балансировщик, который автоматически перемещает chunk'и между шардами.
```
// Включить балансировщик (по умолчанию включен)
sh.startBalancer()

// Настроить окно балансировки (например, ночное время)
db.getSiblingDB("config").settings.updateOne(
  { _id: "balancer" },
  { $set: { activeWindow: { start: "01:00", stop: "05:00" } } },
  { upsert: true }
)

// Проверить статус балансировки
sh.getBalancerState()
```

#### 4.2. Zone Sharding для изоляции популярных категорий
В качестве решения предлагаю создать отдельные зоны для популярных категорий и разместить их на выделенных шардах.
```
// Добавить зоны для популярных категорий
sh.addShardToZone("shard1ReplSet", "high_demand_zone")
sh.addShardToZone("shard2ReplSet", "high_demand_zone")

// Определить диапазоны для популярных категорий
// Используем поле category как префикс шард-ключа
sh.updateZoneKeyRange(
  "somedb.products",
  { category: "Electronics", _id: MinKey() },
  { category: "Electronics", _id: MaxKey() },
  "high_demand_zone"
)

sh.updateZoneKeyRange(
  "somedb.products",
  { category: "Smartphones", _id: MinKey() },
  { category: "Smartphones", _id: MaxKey() },
  "high_demand_zone"
)
```

### 5. Автоматизация реагирования

#### 5.1. Скрипт автоматического реагирования
```
// auto_rebalance.js - скрипт для мониторинга и автоматического реагирования

function checkShardBalance() {
  const chunks = db.getSiblingDB("config").chunks.aggregate([
    { $match: { ns: "somedb.products" } },
    { $group: { _id: "$shard", count: { $sum: 1 } } }
  ]).toArray();
  
  if (chunks.length < 2) return;
  
  const maxCount = Math.max(...chunks.map(c => c.count));
  const minCount = Math.min(...chunks.map(c => c.count));
  const imbalanceRatio = (maxCount - minCount) / maxCount;
  
  if (imbalanceRatio > 0.2) {
    print(`WARNING: Shard imbalance detected: ${imbalanceRatio * 100}%`);
    
    // Проверить, включен ли балансировщик
    const balancerState = sh.getBalancerState();
    if (!balancerState) {
      print("Starting balancer...");
      sh.startBalancer();
    }
  }
}

// Проверять каждые 5 минут
setInterval(checkShardBalance, 300000);
```

## Задание 9. Настройка чтения с реплик и консистентность

### 1. Введение

В MongoDB кластере с репликацией (каждый шард имеет 3 реплики) можно распределять нагрузку на чтение между PRIMARY и SECONDARY репликами. Это позволяет:
- Снизить нагрузку на PRIMARY
- Увеличить пропускную способность чтения
- Обеспечить отказоустойчивость

Однако использование SECONDARY реплик вносит компромисс по **консистентности** из-за асинхронной репликации.

### 2. Определение операций чтения для каждой коллекции

#### 2.1. Коллекция products

| Операция | Тип | Read Preference | Обоснование | Допустимая задержка |
|----------|-----|-----------------|-------------|---------------------|
| **Просмотр товара на странице** (`findOne` по _id) | Чтение | `secondaryPreferred` | Пользователь может видеть не самую свежую цену или описание — это не критично. Задержка репликации < 2-3 секунд допустима. | < 3 сек |
| **Поиск товаров по категориям** (`find` по category + price) | Чтение | `secondaryPreferred` | Поиск и фильтрация не требуют абсолютной актуальности. Допустимо использовать secondary для снижения нагрузки. | < 5 сек |

### 7. Обоснование выбора

#### 7.1. Почему корзина (carts) требует PRIMARY для чтения?
1. **Высокая частота обновлений** — пользователи постоянно добавляют/удаляют товары
2. **Требование консистентности** — пользователь должен видеть актуальное состояние
3. **Бизнес-логика** — добавление товара и отображение корзины — синхронные операции

#### 7.2. Почему товары (products) могут читаться с SECONDARY?
1. **Большой объем чтения** — каталог товаров просматривается значительно чаще, чем обновляется
2. **Некритичность задержки** — цена товара обновленная через 3 секунды после изменения, не влияет на бизнес
3. **Исключение** — проверка остатков при покупке требует PRIMARY

#### 7.3. Почему заказы (orders) могут читаться с SECONDARY?
1. **Исторические данные** — заказы не меняются после оформления
2. **Пользовательский опыт** — задержка в отображении статуса в 2-3 секунды незаметна
3. **Исключение** — оплата требует PRIMARY для проверки актуального статуса


Разработанная стратегия чтения с реплик позволяет:
| Цель | Реализация |
|------|------------|
| **Снижение нагрузки на PRIMARY** | 70% операций чтения направляются на SECONDARY |
| **Гарантия консистентности** | Критические операции (проверка остатков, оплата, корзина) используют PRIMARY |
| **Допустимые задержки** | Определены пороги для каждой операции (1-60 сек в зависимости от важности) |
| **Мониторинг** | Метрики для отслеживания задержки репликации |
**Ключевое решение** - использовать `secondaryPreferred` как read preference по умолчанию, переопределяя на `primary` для критических операций.

## Задание 10. Миграция на Cassandra: модель данных, стратегии репликации и шардирования

### 1. Введение

В условиях пиковой нагрузки 50 000 запросов/сек. MongoDB показала проблемы при масштабировании:
- При добавлении новых шардов происходило полное перераспределение данных
- Это вызывало просадку latency и потребляло ресурсы кластера

Cassandra предлагает архитектуру, свободную от этих недостатков:
- **Leaderless репликация** — нет единой точки отказа
- **Виртуальные узлы (vnodes)** — равномерное распределение данных
- **Добавление узлов** — перераспределяется только часть данных (Token Range)

### 2. Задание 10.1. Выбор сущностей для миграции в Cassandra

#### 2.1. Критерии выбора

| Критерий | Описание |
|----------|----------|
| **Масштабируемость** | Сущности с высокими требованиями к горизонтальному масштабированию |
| **Скорость записи** | Сущности с большим количеством операций записи |
| **Скорость чтения** | Сущности с предсказуемыми паттернами чтения |
| **Целостность данных** | Допустимость EVENTUAL CONSISTENCY |
| **Геораспределенность** | Необходимость мультирегиональной репликации |

#### 2.2. Анализ сущностей

| Сущность | Текущее хранилище | Требования к консистентности | Частота запросов | Целесообразность миграции в Cassandra |
|----------|-------------------|------------------------------|------------------|--------------------------------------|
| **Заказы (orders)** | MongoDB | Strong (финансовые операции) | Высокая | Частично (только история) |
| **Товары (products)** | MongoDB | Strong (остатки) | Очень высокая | Нет (требуют сильной консистентности) |
| **Корзины (carts)** | MongoDB | Strong (актуальность) | Очень высокая | Нет (требуют сильной консистентности) |
| **История заказов** | MongoDB | Eventual | Очень высокая | Да |
| **Пользовательские сессии** | Redis/MongoDB | Eventual | Очень высокая | Да |
| **Аналитика/События** | - | Eventual | Очень высокая | Да |
| **Каталог товаров (чтение)** | MongoDB | Eventual | Очень высокая | Да (только чтение) |

#### 2.3. Решение: какие сущности переносим

| Сущность | Решение | Обоснование |
|----------|---------|-------------|
| **История заказов (order_history)** | Мигрировать | Только запись и чтение по пользователю, eventual consistency допустима |
| **Пользовательские сессии (user_sessions)** | Мигрировать | Высокая нагрузка на запись/чтение, TTL, eventual consistency |
| **События аналитики (analytics_events)** | Мигрировать | Очень высокая нагрузка на запись, eventual consistency |
| **Каталог товаров (product_catalog)** | Мигрировать (read-only) | Денормализованный кэш для чтения |
| **Товары (stock)** | Оставить в MongoDB | Требуют сильной консистентности для остатков |
| **Корзины (carts)** | Оставить в MongoDB | Требуют сильной консистентности |
| **Заказы (orders)** | Оставить в MongoDB | Финансовые операции требуют ACID |

### 3. Задание 10.2. Концептуальная модель данных Cassandra

#### 3.1. Таблица: order_history (история заказов)
```
CREATE TABLE order_history (
    user_id UUID,                      -- PARTITION KEY: пользователь
    order_date DATE,                   -- CLUSTERING KEY: сортировка по дате
    order_id UUID,                     -- CLUSTERING KEY: уникальность
    status TEXT,                       -- статус заказа
    total_amount DECIMAL,              -- сумма заказа
    geo_zone TEXT,                     -- геозона
    items TEXT,                        -- список товаров (JSON)
    PRIMARY KEY ((user_id), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC);
```
**Обоснование:**
| Компонент | Значение | Обоснование |
|-----------|----------|-------------|
| **Partition Key** | `user_id` | Все заказы пользователя в одной партиции — быстрый доступ к истории |
| **Clustering Key** | `order_date`, `order_id` | Сортировка по дате (новые сверху), уникальность |
| **Распределение** | Равномерное по `user_id` | Нет «горячих» партиций при равномерном распределении user_id |
| **Риски** | У популярного пользователя может быть большая партиция | Использовать `ORDER BY DESC` и ограничивать выборку |

#### 3.2. Таблица: user_sessions (пользовательские сессии)
```
CREATE TABLE user_sessions (
    session_id UUID,                   -- PARTITION KEY: сессия
    user_id UUID,                      -- идентификатор пользователя
    created_at TIMESTAMP,              -- время создания
    expires_at TIMESTAMP,              -- время истечения (TTL)
    data TEXT,                         -- данные сессии (JSON)
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400;   -- TTL 24 часа
```
**Обоснование:**
| Компонент | Значение | Обоснование |
|-----------|----------|-------------|
| **Partition Key** | `session_id` | Каждая сессия — отдельная партиция, равномерное распределение |
| **TTL** | 86400 секунд | Автоматическое удаление истекших сессий |
| **Распределение** | Равномерное по `session_id` | Нет «горячих» партиций |

#### 3.3. Таблица: analytics_events (события аналитики)
```
CREATE TABLE analytics_events (
    event_date DATE,                   -- PARTITION KEY: день
    event_type TEXT,                   -- CLUSTERING KEY: тип события
    event_id UUID,                     -- CLUSTERING KEY: уникальность
    user_id UUID,                      -- идентификатор пользователя
    product_id UUID,                   -- идентификатор товара
    event_data TEXT,                   -- данные события (JSON)
    created_at TIMESTAMP,              -- время создания
    PRIMARY KEY ((event_date), event_type, event_id)
);
```
**Обоснование:**
| Компонент | Значение | Обоснование |
|-----------|----------|-------------|
| **Partition Key** | `event_date` | Все события дня в одной партиции — удобно для агрегаций |
| **Clustering Key** | `event_type`, `event_id` | Сортировка по типу события, уникальность |
| **Распределение** | По дням | Естественное распределение, но возможны «горячие» дни (Черная пятница) |
| **Риски** | Пиковые дни создают большие партиции | Использовать bucket-стратегию: `(event_date, event_hour)` |

#### 3.4. Таблица: product_catalog (кэш каталога товаров для чтения)
```
CREATE TABLE product_catalog (
    category TEXT,                     -- PARTITION KEY: категория
    product_id UUID,                   -- CLUSTERING KEY: ID товара
    name TEXT,                         -- наименование
    price DECIMAL,                     -- цена
    attributes TEXT,                   -- атрибуты (JSON)
    PRIMARY KEY ((category), product_id)
);
```
**Обоснование:**
| Компонент | Значение | Обоснование |
|-----------|----------|-------------|
| **Partition Key** | `category` | Поиск товаров по категории — основной паттерн запросов |
| **Clustering Key** | `product_id` | Уникальность внутри категории |
| **Распределение** | По категориям | Риск «горячих» категорий (Электроника) |
| **Риски** | Категория «Электроника» может быть слишком большой | Рассмотреть денормализацию или шардирование по subcategory |

### 5. Задание 10.3. Стратегии обеспечения целостности данных

#### 5.1. Стратегии репликации Cassandra
| Стратегия | Описание | Задержка | Целостность |
|-----------|----------|----------|-------------|
| **Hinted Handoff** | Временное хранение записей для недоступных узлов | Низкая | Слабая (временная) |
| **Read Repair** | Проверка и исправление рассинхронизации при чтении | Средняя | Сильная |
| **Anti-Entropy Repair** | Периодическая фоновая синхронизация (Merkle trees) | Высокая | Полная |

#### 5.2. Выбор стратегий по сущностям
| Таблица | Hinted Handoff | Read Repair | Anti-Entropy Repair | Обоснование |
|---------|----------------|-------------|---------------------|-------------|
| **order_history** | Включена | Включена (10%) | Ежедневно | История заказов должна быть точной, но допустима небольшая задержка |
| **user_sessions** | Включена | Отключена | Ежедневно | Сессии имеют TTL, допускают eventual consistency |
| **analytics_events** | Включена | Отключена | Ежечасно | Аналитика допускает eventual, важна скорость записи |
| **product_catalog** | Включена | Включена (25%) | Ежедневно | Каталог должен быть консистентным, но eventual допустима |

#### 5.3. Конфигурация стратегий
```
-- Создание keyspace с репликацией
CREATE KEYSPACE mobile_world
WITH REPLICATION = {
    'class': 'NetworkTopologyStrategy',
    'datacenter1': 3,           -- 3 реплики в дата-центре
    'datacenter2': 2            -- 2 реплики в другом дата-центре
};

-- Настройка таблиц с параметрами согласованности
-- order_history: высокая согласованность
CREATE TABLE order_history (...)
WITH read_repair_chance = 0.1        -- 10% запросов на чтение запускают read repair
AND dclocal_read_repair_chance = 0.1
AND gc_grace_seconds = 86400;        -- 24 часа для восстановления

-- user_sessions: скорость важнее согласованности
CREATE TABLE user_sessions (...)
WITH read_repair_chance = 0.0        -- отключен read repair
AND gc_grace_seconds = 86400
AND default_time_to_live = 86400;    -- TTL 24 часа

-- analytics_events: максимальная скорость записи
CREATE TABLE analytics_events (...)
WITH read_repair_chance = 0.0        -- отключен read repair
AND gc_grace_seconds = 3600;         -- 1 час для восстановления

-- product_catalog: умеренная согласованность
CREATE TABLE product_catalog (...)
WITH read_repair_chance = 0.25       -- 25% запросов запускают read repair
AND gc_grace_seconds = 86400;
```

### 6. Уровни согласованности (Consistency Levels)
| Операция | Таблица | CL | Обоснование |
|----------|---------|----|-------------|
| **Запись** | order_history | QUORUM | История заказов должна быть записана на большинство узлов |
| **Запись** | user_sessions | ONE | Скорость критична, потеря сессии не страшна |
| **Запись** | analytics_events | ONE | Максимальная скорость записи |
| **Запись** | product_catalog | LOCAL_QUORUM | Баланс между скоростью и надежностью |
| **Чтение** | order_history | QUORUM | Точность истории заказов |
| **Чтение** | user_sessions | ONE | Скорость критична |
| **Чтение** | analytics_events | ONE | Аналитика допускает неточности |
| **Чтение** | product_catalog | LOCAL_QUORUM | Баланс скорости и точности |

### 8. Метрики мониторинга для Cassandra
| Метрика | Описание | Порог | Действие |
|---------|----------|-------|----------|
| **Write latency** | Задержка записи | > 50ms | Проверить нагрузку на узлы, добавить узлы |
| **Read latency** | Задержка чтения | > 100ms | Проверить индексы, оптимизировать запросы |
| **Pending compactions** | Ожидающие компрессии | > 10 | Увеличить ресурсы или время окна |
| **Hinted handoff backlog** | Накопленные подсказки | > 1000 | Проверить недоступные узлы |
| **Read repair chance** | Вероятность read repair | > 50% | Снизить вероятность, если нагрузка высокая |
| **Partition size** | Размер партиции | > 100MB | Пересмотреть partition key |

### 9. Заключение по Заданию 10
| Требование | Реализация |
|------------|------------|
| **Высокая отказоустойчивость** | Leaderless репликация с NetworkTopologyStrategy |
| **Быстрое горизонтальное масштабирование** | Virtual nodes (vnodes) — добавление узлов без полного перераспределения |
| **Равномерное распределение данных** | Partition key на основе user_id, session_id, event_date |
| **Обработка «горячих» партиций** | Bucket-стратегия для пиковых дней (event_date + hour) |
| **Целостность данных** | Read Repair для критических данных, Hinted Handoff для временной доступности, Anti-Entropy Repair для фоновой синхронизации |

**Ключевые решения:**
- **History & Analytics** — мигрируем в Cassandra (high write throughput, eventual consistency)
- **Critical data (stock, carts, orders)** — оставляем в MongoDB (strong consistency, ACID)
- **Read repair** — включаем для `order_history` и `product_catalog` (10-25%)
- **TTL** — используем для user_sessions (24h)
- **Consistency Level** — QUORUM для критических данных, ONE для аналитики
